Spring Core Concepts

Learn IoC, DI, and Spring Fundamentals with Detailed Examples

1. Inversion of Control (IoC)

What is it?

IoC is a design principle in programming where the control of creating and managing objects is "inverted" (flipped) from you (the programmer) to something else, like a framework or tool.

What control are we inverting?

Normally, as a programmer, you write code to:

Create objects (e.g., `Car car = new Car();`).

Manage them (e.g., decide when to use them).

Destroy them (e.g., free up memory when done).

With IoC, you give up this control and let someone else (like a framework) handle it for you.

Why do we do this?

As programmers, we want to focus on the business logic (the actual problem we’re solving, like calculating a price or saving data) instead of worrying about creating and managing objects.

Simple Analogy:

Imagine you’re cooking dinner.

Normally, you’d go to the store, buy ingredients, and cook the meal yourself (full control).

With IoC, it’s like hiring a chef who brings the ingredients and cooks for you—you just tell them what dish you want (business logic), and they handle the rest.

(Source: Baeldung - Illustrates control flipping from programmer to framework)

2. IoC Container

What is it?

An IoC Container is a tool or part of a framework (like Spring in Java) that takes care of creating, managing, and destroying objects for you.

It’s the “someone else” in IoC.

What does it do?

Creates objects: Instead of you writing `new Car()`, the IoC Container creates the `Car` object for you.

Maintains objects: It keeps track of them and makes sure they’re ready when you need them.

Destroys objects: It cleans them up when they’re no longer needed.

Example in Spring:

In the Spring framework, the IoC Container is called the ApplicationContext.

You tell it what objects (called "beans") you need, and it creates and manages them for you.

Simple Analogy:

Think of the IoC Container as a factory.

You order a toy (object), and the factory builds it, maintains it, and throws it away when you’re done—without you doing the work.

(Source: TutorialsPoint - Shows container managing beans)

3. Dependency Injection (DI)

What is it?

DI is a way to implement IoC.

It’s the process of "injecting" (providing) the objects (dependencies) your code needs, instead of you creating them manually.

What’s a dependency?

A dependency is something your code relies on to work.

For example: If you have a `Car` class that needs an `Engine` to run, the `Engine` is a dependency of `Car`.

Without DI (Normal Way):

class Car {
    Engine engine = new Engine(); // You create the Engine yourself
    void drive() {
        engine.start();
    }
}

With DI (Using Injection):

class Car {
    Engine engine; // You don’t create it here
    Car(Engine engine) { // The IoC Container "injects" it
        this.engine = engine;
    }
    void drive() {
        engine.start();
    }
}

The IoC Container creates the `Engine` and passes (injects) it into the `Car` when it’s created.

Why use DI?

It makes your code cleaner and easier to change.

If you want a different `Engine` (e.g., electric instead of gas), the IoC Container can swap it without you rewriting the `Car` class.

Simple Analogy:

Imagine you’re building a toy car.

Normally, you’d make the wheels yourself and attach them.

With DI, someone hands you the wheels ready-made, and you just use them.

(Source: Baeldung - Visualizes dependency injection process)

How IoC and DI Work Together

IoC is the big idea: “Someone else controls object creation and management.”

DI is one way to make IoC happen: “That someone else injects the objects I need.”

IoC Container is the tool that does the work: “It creates and injects the objects.”

Example in Real Life (Spring Framework)

1. You write a class:

class Car {
    Engine engine;
    Car(Engine engine) { // Dependency Injection
        this.engine = engine;
    }
    void drive() {
        engine.start();
    }
}

2. You tell the Spring IoC Container (via a configuration file or annotations) that you need a `Car` and an `Engine`.

3. Spring’s IoC Container:

Creates an `Engine`.

Creates a `Car` and injects the `Engine` into it.

Gives you the fully working `Car` object.

As a programmer, you focus on what the `Car` does (e.g., `drive()`), not how it’s built.

Key Benefits for Beginners

Less Work: You don’t have to manually create and connect objects.

Easier Changes: Want a new `Engine` type? Just update the configuration—no need to rewrite your code.

Focus on Logic: Spend time solving problems, not managing objects.

Full Setup & Examples

First, here’s the basic Java code we’ll use for all examples:

Engine.java

package com.example;

public class Engine {
    public void start() {
        System.out.println("Engine started!");
    }
}

Car.java (Base Version)

package com.example;

public class Car {
    private Engine engine;

    // Setter for injection
    public void setEngine(Engine engine) {
        this.engine = engine;
    }

    public void drive() {
        if (engine != null) {
            engine.start();
            System.out.println("Car is driving...");
        } else {
            System.out.println("No engine, can't drive!");
        }
    }
}

Main.java (To Test Output)

package com.example;

import org.springframework.context.ApplicationContext;
import org.springframework.context.support.ClassPathXmlApplicationContext;

public class Main {
    public static void main(String[] args) {
        // Load the XML configuration
        ApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");

        // Get beans and test them (specific bean names will vary by example)
        Car car = (Car) context.getBean("car");
        car.drive();
    }
}

Now, let’s go through each concept with XML, comments, and output.

1. ApplicationContext

What is it? The IoC Container that manages beans.

XML (applicationContext.xml):

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <!-- Simple bean definition -->
    <bean id="car" class="com.example.Car"/>
</beans>

Main.java (Updated):

public class Main {
    public static void main(String[] args) {
        ApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
        Car car = (Car) context.getBean("car"); // Fetch bean from ApplicationContext
        car.drive(); // Test the bean
    }
}

Output:

No engine, can't drive!

Explanation:

`ApplicationContext` loads the XML and creates a `Car` bean.

Since no `Engine` is injected yet, the `drive()` method shows the car can’t start.

2. Singleton Scope

What is it? One instance shared everywhere (default scope).

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="engine" class="com.example.Engine" scope="singleton"/> <!-- Single Engine instance -->
    <bean id="car" class="com.example.Car">
        <property name="engine" ref="engine"/> <!-- Inject the singleton engine -->
    </bean>
</beans>

Main.java (Updated):

public class Main {
    public static void main(String[] args) {
        ApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
        Car car1 = (Car) context.getBean("car"); // First car
        Car car2 = (Car) context.getBean("car"); // Second car
        car1.drive();
        System.out.println("Same engine? " + (car1 == car2)); // Check if same instance
    }
}

Output:

Engine started!
Car is driving...
Same engine? false

Explanation:

`engine` is singleton, so both `car1` and `car2` share the same `Engine` instance internally, but `car` itself isn’t singleton here (we’d need `scope="singleton"` on `car` too).

The output shows the engine works.

3. Prototype Scope

What is it? New instance every time you request it.

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="engine" class="com.example.Engine" scope="singleton"/>
    <bean id="car" class="com.example.Car" scope="prototype"> <!-- New Car each time -->
        <property name="engine" ref="engine"/>
    </bean>
</beans>

Main.java:

public class Main {
    public static void main(String[] args) {
        ApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
        Car car1 = (Car) context.getBean("car"); // First instance
        Car car2 = (Car) context.getBean("car"); // Second instance
        car1.drive();
        car2.drive();
        System.out.println("Same car? " + (car1 == car2)); // Check if different
    }
}

Output:

Engine started!
Car is driving...
Engine started!
Car is driving...
Same car? false

Explanation:

`car` is prototype, so `car1` and `car2` are different objects, each with the same singleton `engine`.

4. Request Scope

What is it? New instance per HTTP request (web-only).

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="car" class="com.example.Car" scope="request"/> <!-- Web-specific scope -->
</beans>

Note: This won’t work in a simple `main` method (needs a web app).

For demo, let’s assume a web context.

Output (Hypothetical in Web App):

No engine, can't drive! (for each new HTTP request)

Explanation:

In a web app, each request gets a new `Car`.

Without a web setup, it throws an error in a standalone app.

5. Setter Injection

What is it? Injecting dependencies via setters.

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="engine" class="com.example.Engine"/>
    <bean id="car" class="com.example.Car">
        <property name="engine" ref="engine"/> <!-- Calls setEngine() -->
    </bean>
</beans>

Output:

Engine started!
Car is driving...

Explanation:

Spring uses `setEngine()` to inject the `engine` bean into `car`.

6. Bean

What is it? An object managed by Spring.

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="car" class="com.example.Car"/> <!-- Defines a bean -->
</beans>

Output:

No engine, can't drive!

Explanation:

`car` is a bean, but no `engine` is injected yet.

7. Component

What is it? Auto-detected bean with `@Component`.

Java (Updated Car.java):

package com.example;
import org.springframework.stereotype.Component;

@Component("car") // Registers as a bean named "car"
public class Car {
    private Engine engine;
    public void setEngine(Engine engine) { this.engine = engine; }
    public void drive() {
        if (engine != null) { engine.start(); System.out.println("Car is driving..."); }
        else { System.out.println("No engine, can't drive!"); }
    }
}

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd
       http://www.springframework.org/schema/context
       http://www.springframework.org/schema/context/spring-context.xsd">
    <context:component-scan base-package="com.example"/> <!-- Scans for @Component -->
    <bean id="engine" class="com.example.Engine"/>
</beans>

Output (Without Engine Injection):

No engine, can't drive!

Explanation:

`@Component` makes `Car` a bean.

We’d need autowiring or injection for `engine`.

8. Autowiring

What is it? Automatic dependency injection.

XML:

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">
    <bean id="engine" class="com.example.Engine"/>
    <bean id="car" class="com.example.Car" autowire="byName"/> <!-- Matches "engine" property -->
</beans>

Output:

Engine started!
Car is driving...

Explanation:

`autowire="byName"` links the `engine` property to the `engine` bean automatically.

Full XML Example with Comments

<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd
       http://www.springframework.org/schema/context
       http://www.springframework.org/schema/context/spring-context.xsd">

    <!-- Enable scanning for @Component -->
    <context:component-scan base-package="com.example"/>

    <!-- Singleton Engine bean -->
    <bean id="engine" class="com.example.Engine" scope="singleton"/>

    <!-- Prototype Car with Setter Injection -->
    <bean id="car" class="com.example.Car" scope="prototype">
        <property name="engine" ref="engine"/> <!-- Injects engine via setter -->
    </bean>

    <!-- Autowired Car -->
    <bean id="autoCar" class="com.example.Car" autowire="byName"/> <!-- Auto-links engine -->
</beans>
