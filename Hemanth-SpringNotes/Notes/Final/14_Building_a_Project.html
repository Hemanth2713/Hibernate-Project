Job Portal Web Application Notes

Comprehensive guide to building a job portal using Spring Boot and JSP

Project Overview

The goal is to build a Job Portal Web Application where:

  • Employers can post job openings with details like job profile, description, experience required, and tech stack.
  • Job Seekers can view all available jobs and their requirements.
  • The application uses Spring Boot for the backend, JSP for views, and follows a layered architecture (Controller, Service, Repository).
  • Data is stored in-memory using an ArrayList (no database for simplicity, but designed to be extensible for database integration).
  • The frontend is styled with Bootstrap and custom CSS, but the focus is on backend logic.

Features

  1. Home Page: Displays navigation links to view jobs or add a job.
  2. View All Jobs: Lists all job postings with details (profile, description, experience, tech stack).
  3. Add Job: Allows employers to submit a new job posting via a form.
  4. Success Page: Confirms successful job posting with details.

Real-Time Example

  • LinkedIn: A job portal where companies post jobs, and users search/apply for them.
  • Naukri.com: Similar functionality where employers post job requirements, and job seekers browse listings.
  • This project mimics a simplified version of such platforms, focusing on CRUD-like operations (Create and Read) without a database.

Prerequisites

Technologies Used:

  • Spring Boot: Simplifies Spring configuration and setup.
  • Spring MVC: Handles web requests and responses.
  • JSP (JavaServer Pages): Used for rendering views.
  • Lombok: Reduces boilerplate code (getters, setters, constructors).
  • Maven: Dependency management.
  • Bootstrap: Frontend styling.
  • Tomcat: Embedded server for running the application.

Tools:

  • IntelliJ IDEA Community Edition: IDE for development.
  • Spring Initializr (start.spring.io): To generate the project.
  • Java 21: Programming language version.

Dependencies (from pom.xml):

  • spring-boot-starter-web: For building web applications.
  • lombok: For reducing boilerplate code.
  • tomcat-jasper: For JSP rendering.
  • jakarta.servlet.jsp.jstl: JSTL for JSP tag libraries.
  • jakarta.servlet.jsp.jstl-api: JSTL API for JSP.

Application Architecture

Layered Architecture

The application follows the MVC (Model-View-Controller) pattern with a layered approach:

  1. Controller Layer (JobController): Handles HTTP requests, interacts with the service layer, and returns views or data.
  2. Service Layer (JobService): Contains business logic and communicates with the repository.
  3. Repository Layer (JobRepo): Manages data storage (in-memory ArrayList for now, extensible to a database).
  4. Model (JobPost): Represents the data structure for a job posting.
  5. View (JSP files): Renders the UI using JSP and JSTL.

Data Flow

  • Client Request → Controller → Service → Repository → Data Storage (and vice versa for responses).
  • DTO (Data Transfer Object): The JobPost class acts as a DTO, transferring data between layers.

import lombok.Data;

// @Data generates getters, setters, toString, equals, and hashCode automatically
@Data

@NoArgsConstructor

@AllArgsConstructor

@Component

    // Detailed description of the job
    private String postDesc;
    // Required years of experience
    private Integer reqExperience;
    // List of required technologies/skills
    private List<String> postTechStack;
}

Explanation:

  • Lombok Annotations:
    • @Data: Generates getters, setters, toString, equals, and hashCode.
    • @NoArgsConstructor: Creates a default constructor.
    • @AllArgsConstructor: Creates a constructor with all fields.
  • @Component: Registers JobPost as a Spring bean, injectable into other components.
  • Fields:

2. Repository Layer: JobRepo.java

The JobRepo class simulates a database by storing job posts in an in-memory ArrayList.

Code: src/main/java/com/telusko/jobapp/repo/JobRepo.java 

package com.telusko.jobapp.repo;

import com.telusko.jobapp.model.JobPost;
import org.springframework.stereotype.Repository;

import java.util.ArrayList;
import java.util.Arrays;
import java.util.List;

@Repository // Marks this class as a Spring repository
public class JobRepo {
    // In-memory storage for job posts (simulates a database)
    private List<JobPost> jobs = new ArrayList<>(Arrays.asList(
            // Pre-populated dummy data
            new JobPost(1, "Java Developer", "Must have good experience in core Java and advanced Java", 2,
                    List.of("Java", "Spring", "Hibernate")),
            new JobPost(2, "Frontend Developer", "Experience in building responsive web applications", 3,
                    List.of("HTML", "CSS", "JavaScript", "React")),
            new JobPost(3, "Data Scientist", "Strong knowledge in machine learning and statistics", 4,
                    List.of("Python", "TensorFlow", "SQL")),
            new JobPost(4, "DevOps Engineer", "Experience in CI/CD pipelines and containerization", 3,
                    List.of("Docker", "Kubernetes", "AWS")),
            new JobPost(5, "Full Stack Developer", "Expertise in both frontend and backend technologies", 5,
                    List.of("Java", "JavaScript", "Node.js", "Angular"))
    ));

    // Returns all job posts
    public List<JobPost> getAllJobs() {
        return jobs;
    }

    // Adds a new job post to the list
    public void addJob(JobPost job) {
        jobs.add(job);
        // Print the list for debugging (temporary)
        System.out.println(jobs);
    }
}

Output: This Java class does not produce direct output when compiled, but it initializes an ArrayList with five predefined JobPost objects. When addJob() is called, it adds a new job and prints the updated list to the console for debugging. For example, adding a new job with postId=6 would print the list of six jobs, including the new one, to the console.

Example Console Output (if a new job is added): 

[JobPost(postId=1, postProfile=Java Developer, ...), JobPost(postId=2, postProfile=Frontend Developer, ...), ..., JobPost(postId=6, postProfile=New Job, ...)]

Explanation:

  • @Repository: Marks this class as a Spring data access component.
  • jobs: An ArrayList initialized with dummy job posts to simulate a database.
  • Methods:
    • getAllJobs(): Returns the list of all job posts.
    • addJob(JobPost): Adds a new job post to the list and prints the list for debugging.
  • Key Point: Using new ArrayList<>(Arrays.asList(...)) ensures the list is mutable, unlike List.of(), which creates an immutable list.
  • Real-Time Example: In a real job portal, this layer would interact with a database (e.g., MySQL, MongoDB) using JPA or JDBC to store and retrieve job postings.
  • Extensibility: Replace ArrayList with database operations (e.g., using Spring Data JPA) in the future.

3. Service Layer: JobService.java

The JobService class contains business logic and acts as an intermediary between the controller and repository.

Code: src/main/java/com/telusko/jobapp/service/JobService.java 

package com.telusko.jobapp.service;

import com.telusko.jobapp.model.JobPost;
import com.telusko.jobapp.repo.JobRepo;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

import java.util.List;

@Service // Marks this class as a Spring service
public class JobService {
    // Injects the JobRepo dependency
    @Autowired
    private JobRepo repo;

    // Retrieves all job posts from the repository
    public List<JobPost> getAllJobs() {
        return repo.getAllJobs();
    }

    // Adds a new job post via the repository
    public void addJob(JobPost job) {
        repo.addJob(job);
    }
}

Output: This Java class does not produce direct output. It provides methods to interact with the repository. getAllJobs() returns the list of job posts, and addJob() adds a new job, which may trigger console output in JobRepo (as shown above). The service layer is invoked by the controller to handle business logic.

Explanation:

  • @Service: Marks this class as a Spring service component.
  • @Autowired: Injects the JobRepo instance, enabling communication with the repository.
  • Methods:
    • getAllJobs(): Delegates to JobRepo to fetch all job posts.
    • addJob(JobPost): Delegates to JobRepo to add a new job post.
  • Purpose: Isolates business logic (e.g., validation, transformation) from data access and controller logic.
  • Real-Time Example: In a job portal, the service layer might validate job post data (e.g., ensure experience is positive) before saving it.

4. Controller Layer: JobController.java

The JobController handles HTTP requests, interacts with the service layer, and returns JSP views.

Code: src/main/java/com/telusko/jobapp/JobController.java 

package com.telusko.jobapp;

import com.telusko.jobapp.model.JobPost;
import com.telusko.jobapp.service.JobService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PostMapping;

import java.util.List;

@Controller // Marks this class as a Spring MVC controller
public class JobController {
    // Injects the JobService dependency
    @Autowired
    private JobService service;

    // Handles GET requests for "/" and "/home", returns home.jsp
    @GetMapping({"/", "home"})
    public String home() {
        return "home"; // Resolves to /views/home.jsp
    }

    // Handles GET requests for "/addjob", returns addjob.jsp
    @GetMapping("addjob")
    public String addJob() {
        return "addjob"; // Resolves to /views/addjob.jsp
    }

    // Handles POST requests for "/handleForm", processes form data, and returns success.jsp
    @PostMapping("handleForm")
    public String handleForm(JobPost jobPost, Model model) {
        // Add the job post to the service layer
        service.addJob(jobPost);
        // Add the job post to the model for success.jsp
        model.addAttribute("jobPost", jobPost);
        return "success"; // Resolves to /views/success.jsp
    }

    // Handles GET requests for "/viewalljobs", fetches all jobs, and returns viewalljobs.jsp
    @GetMapping("viewalljobs")
    public String viewJobs(Model model) {
        // Fetch all job posts from the service layer
        List<JobPost> jobs = service.getAllJobs();
        // Add the job posts to the model for viewalljobs.jsp
        model.addAttribute("jobPosts", jobs);
        return "viewalljobs"; // Resolves to /views/viewalljobs.jsp
    }
}

Output: This Java class does not produce direct console output but defines HTTP endpoints that return JSP views. The methods map to the following URLs and outcomes:

  • GET / or /home: Renders home.jsp (home page).
  • GET /addjob: Renders addjob.jsp (job posting form).
  • POST /handleForm: Processes form data, adds a job, and renders success.jsp (success page).
  • GET /viewalljobs: Renders viewalljobs.jsp (list of all jobs).

The actual output is the rendered JSP pages, described in their respective sections below.

Explanation:

  • @Controller: Marks this class as a Spring MVC controller.
  • @Autowired: Injects the JobService instance.
  • Mappings:
    • @GetMapping({"/", "home"}): Maps home() to "/" and "/home", returning home.jsp.
    • @GetMapping("addjob"): Maps addJob() to "/addjob", returning addjob.jsp.
    • @PostMapping("handleForm"): Maps handleForm() to "/handleForm", processes form data, and returns success.jsp.
    • @GetMapping("viewalljobs"): Maps viewJobs() to "/viewalljobs", fetches all jobs, and returns viewalljobs.jsp.
  • Model: Used to pass data (e.g., jobPost, jobPosts) to JSP views.
  • Real-Time Example: In a job portal, controllers handle user actions like submitting a job form or viewing job listings, similar to this setup.

    • @SpringBootApplication: Combines @Configuration, @EnableAutoConfiguration, and @ComponentScan.

2. Spring MVC

  • Components:
    • Controller: Handles HTTP requests (@Controller, @GetMapping, @PostMapping).
    • Model: Passes data to views (Model interface).
    • View: Renders the UI (JSP).
  • Request Handling:
    • GET: Fetches data (e.g., /viewalljobs).
    • POST: Submits data (e.g., /handleForm).

3. JSP and JSTL

  • JSP: Server-side technology for dynamic web pages.
  • JSTL: Provides tags like for looping and conditionals.
  • EL: Accesses model attributes (e.g., ${jobPost.postId}).
  • Dependencies: tomcat-jasper, jakarta.servlet.jsp.jstl.

4. Lombok

  • Reduces boilerplate code using annotations (@Data, @NoArgsConstructor, @AllArgsConstructor).
  • Example: In JobPost, Lombok generates getters/setters for postId, postProfile, etc.

5. Layered Architecture

  • Controller: Handles user input and orchestrates responses.
  • Service: Encapsulates business logic.
  • Repository: Manages data access.
  • DTO: JobPost transfers data between layers.

6. Dependency Injection

  • @Autowired: Injects dependencies (e.g., JobService into JobController, JobRepo into JobService).
  • Purpose: Promotes loose coupling and testability.

7. In-Memory Storage

  • Uses ArrayList to store job posts, simulating a database.
  • Limitation: Data is lost on application restart.
  • Future Enhancement: Integrate with a database using Spring Data JPA.

8. Bootstrap

  • Provides responsive UI components (navbar, forms, tables).
  • CDN: Used to include Bootstrap CSS and JS.

9. Maven

  • Manages dependencies and builds the project.
  • pom.xml: Defines project metadata and dependencies.

Debugging and Common Issues

  1. 404 Error (Page Not Found):
    • Cause: Incorrect URL mapping or missing JSP file.
    • Solution: Verify @GetMapping/@PostMapping URLs and JSP file names.
  2. 500 Error (Internal Server Error):
    • Cause: Missing model attributes or incorrect package imports.
    • Solution: Ensure model attributes (e.g., jobPost) are added and packages are correct (e.g., com.telusko.jobapp.model.JobPost).
  3. Immutable List Issue:
    • Cause: Using List.of() creates an immutable list, causing errors when adding jobs.
    • Solution: Use new ArrayList<>(Arrays.asList(...)) for a mutable list.
  4. Lombok Not Working:
    • Cause: Annotation processing not enabled in IntelliJ.
    • Solution: Enable annotation processing in IntelliJ settings.

Running the Application

  1. Clone or Open the Project:
    • Open the project in IntelliJ IDEA Community Edition or another IDE.
    • If cloning from a repository, use: git clone <repository-url>.
  2. Verify Dependencies:
    • Ensure all dependencies in pom.xml are downloaded by running mvn install or letting IntelliJ sync the project.
    • Check for spring-boot-starter-web, lombok, tomcat-jasper, and JSTL dependencies.
  3. Configure Java:
    • Set the project SDK to Java 21 in IntelliJ (File > Project Structure > Project SDK).
    • Ensure the Java version matches the one specified in pom.xml (<java.version>21</java.version>).
  4. Run the Application:
    • Locate JobappApplication.java in src/main/java/com/telusko/jobapp.
    • Right-click and select Run 'JobappApplication' in IntelliJ, or use the play button.
    • Alternatively, run via Maven: mvn spring-boot:run from the project root.
    • The embedded Tomcat server starts on http://localhost:8080.
  5. Access the Application:
    • Open a browser and navigate to http://localhost:8080.
    • The home page (home.jsp) should load, displaying the navigation bar and welcome message.
    • Test the following URLs:
      • /home: Home page.
      • /addjob: Form to add a new job.
      • /viewalljobs: List of all jobs.
  6. Test the Application:
    • Add a job via /addjob and submit the form to /handleForm.
    • Verify the job appears on /viewalljobs.
    • Check the console for debugging output from JobRepo when adding jobs.

Note: Data is stored in-memory, so restarting the application resets the job list to the predefined entries in JobRepo.

Output: After running the application, the browser at http://localhost:8080 displays the home page with a navigation bar and welcome message. Navigating to /viewalljobs shows a table of jobs, and submitting a job via /addjob redirects to success.jsp with job details.
