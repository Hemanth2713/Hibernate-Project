Web Application Development Notes

Comprehensive guide to .jar, .war, Embedded Tomcat, Jakarta Servlet, JSP, and MVC

1. Core Technologies

1.1 .jar (Java Archive)

Explanation: A .jar file is a compressed archive that bundles Java class files, resources (e.g., images, properties files), and metadata (e.g., MANIFEST.MF). It’s used for libraries, standalone applications, or dependencies in a project.
Purpose: Simplifies distribution and reuse of Java code.
Example: A library like gson.jar for JSON processing can be included in a project. A standalone app (e.g., a command-line tool) can be packaged as myapp.jar and run with java -jar myapp.jar.
Code Example: No direct code needed, but a .jar might be included in a project’s pom.xml (Maven): 
<!-- Maven dependency for Gson -->
<dependency>
    <groupId>com.google.code.gson</groupId>
    <artifactId>gson</artifactId>
    <version>2.10.1</version>
</dependency>
Note: In web apps, .jar files are often dependencies in the WEB-INF/lib folder of a .war.

1.2 .war (Web Archive)

Explanation: A .war file is a specialized .jar designed for web applications. It contains servlets, JSPs, HTML, CSS, JavaScript, and configuration files (e.g., web.xml) needed to run a web app on a server like Tomcat.
Purpose: Packages the entire web application for deployment.
Example: A student management system might be packaged as studentapp.war and deployed to Tomcat. When deployed, the server extracts the .war and serves its contents (e.g., JSP pages, servlets).
Structure:
studentapp.war/
├── META-INF/
├── WEB-INF/
│   ├── web.xml          (Configuration)
│   ├── lib/             (Dependencies, e.g., gson.jar)
│   ├── classes/         (Compiled Java classes)
├── index.jsp            (Web pages)
├── css/                 (Static resources)
Note: Unlike .jar, .war files are tailored for web servers and include web-specific resources.

1.3 Embedded Tomcat

Explanation: Embedded Tomcat is a version of the Tomcat server bundled within the application, allowing it to run without a separate server installation. It’s started programmatically or via a build tool.
Purpose: Simplifies development and deployment by embedding the server in the app.
Example: In a Maven project, include Tomcat as a dependency and run the app. The app starts a web server on a port (e.g., http://localhost:8080). Ideal for microservices or standalone web apps.
Code Example: Minimal setup for embedded Tomcat in a Java main method: 
import org.apache.catalina.startup.Tomcat;

public class Main {
    public static void main(String[] args) throws Exception {
        Tomcat tomcat = new Tomcat();
        tomcat.setPort(8080);
        tomcat.addWebapp("", new File("src/main/webapp").getAbsolutePath());
        tomcat.start();
        tomcat.getServer().await();
    }
}
Note: In frameworks like Spring Boot, embedded Tomcat is included by default with spring-boot-starter-web, making setup even easier.

1.4 Jakarta Servlet

Explanation: A Jakarta Servlet is a Java class that handles HTTP requests (e.g., GET, POST) and generates responses. It’s the backbone for processing client inputs in a web application, often acting as the controller in MVC.
Purpose: Manages server-side logic, routing, and coordination between model and view.
Example: A servlet mapped to /add accepts two numbers from a form, computes their sum, and forwards the result to a JSP. A servlet mapped to /getStudent retrieves student data for a given ID.
Code Example: See MVC section below for servlet code.
Note: Jakarta Servlets (part of Jakarta EE, formerly Java EE) replace manual request handling with a standardized API.

1.5 JSP (JavaServer Pages)

Explanation: JSP is a technology for creating dynamic web pages. It combines HTML with Java code (via scriptlets, Expression Language, or JSTL) to render data dynamically, typically used as the view in MVC.
Purpose: Generates the user interface based on model data.
Example: A JSP displays a student’s details (e.g., name, ID) fetched by a servlet. A JSP shows the result of adding two numbers.
Code Example: See MVC section below for JSP code.
Note: Modern apps may use alternatives like Thymeleaf, but JSP is still common in legacy or simple projects.

2. MVC Architecture (Model-View-Controller)

2.1 Overview

Explanation: MVC is a design pattern that organizes a web application into three components:
- Model: Represents data and business logic (e.g., a Student object or computation result).
- View: Renders the user interface (e.g., JSP, HTML).
- Controller: Handles client requests, coordinates model and view (e.g., servlet).
Purpose: Separates concerns for modularity, maintainability, and scalability.
Example: A user requests student details for ID 101. The controller fetches data, the model holds it, and the view displays it. A user submits two numbers to add. The controller processes the input, the model stores the result, and the view shows it.

2.2 Workflow

1. Client Request: Sent via browser (e.g., http://example.com/getStudent?id=101 or http://example.com/add?num1=5&num2=3).
2. Controller: A servlet receives the request, extracts parameters (e.g., id, num1, num2). Queries the database or performs calculations. Creates a model object to hold data. Forwards the model to the view.
3. Model: A Java object (e.g., Student) or simple data (e.g., sum of numbers). Encapsulates data and sometimes logic (e.g., validation).
4. View: A JSP (or other view technology) retrieves data from the model. Formats it into HTML and sends it to the client.
5. Response: Client sees the rendered page (e.g., student details or sum).

2.3 Detailed Examples with Code

Example 1: Fetch Student Details
Scenario: A client requests details for a student with ID 101.
Request URL: http://example.com/getStudent?id=101
1. Controller (Servlet): A servlet mapped to /getStudent extracts the ID, fetches data (mocked here for simplicity), and forwards to a JSP. 
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.io.IOException;

@WebServlet("/getStudent")
public class StudentController extends HttpServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {
        int id = Integer.parseInt(request.getParameter("id"));
        // Mock database call
        Student student = new Student(id, "John Doe", 20);
        request.setAttribute("student", student); // Pass model to view
        request.getRequestDispatcher("studentDetails.jsp").forward(request, response);
    }
}
2. Model: A Student class to hold data. 
public class Student {
    private int id;
    private String name;
    private int age;

    public Student(int id, String name, int age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }

    // Getters and setters
    public int getId() { return id; }
    public String getName() { return name; }
    public int getAge() { return age; }
}
3. View (JSP): studentDetails.jsp displays the student’s details.

4. Response: The client sees:

Name: John Doe
Age: 20
Example 2: Add Two Numbers
Scenario: A client submits two numbers (e.g., 5 and 3) to add.
Request URL: http://example.com/add?num1=5&num2=3
1. Controller (Servlet): A servlet mapped to /add extracts numbers, computes the sum, and forwards to a JSP. 
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.io.IOException;

@WebServlet("/add")
public class AddController extends HttpServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {
        int num1 = Integer.parseInt(request.getParameter("num1"));
        int num2 = Integer.parseInt(request.getParameter("num2"));
        int sum = num1 + num2; // Model data
        request.setAttribute("result", sum); // Pass model to view
        request.getRequestDispatcher("result.jsp").forward(request, response);
    }
}
2. Model: In this case, the model is simple: the sum integer. No separate class is needed for such a basic operation, but a class could be used for complex models.
3. View (JSP): result.jsp displays the sum. 
<%@ page language="java" contentType="text/html; charset=UTF-8" pageEncoding="UTF-8"%>



    Addition Result


    
Addition Result

    
Sum: ${result}
4. Response: The client sees: 
Addition Result
Sum: 8

3. How It All Ties Together

Explanation: The web application is packaged as a .war file, containing servlets, JSPs, and resources. Embedded Tomcat runs the app, serving requests on a port (e.g., 8080).
Requests are handled by MVC:
- Controller (servlets like StudentController, AddController) processes inputs.
- Model (e.g., Student object or sum) holds data.
- View (JSPs like studentDetails.jsp, result.jsp) renders the UI.
Example Deployment: Package the app as myapp.war. Run with embedded Tomcat: java -jar myapp.jar (if using Spring Boot) or a custom main class. Access at http://localhost:8080/getStudent?id=101 or http://localhost:8080/add?num1=5&num2=3.
Benefits: MVC separates concerns for clean code. Embedded Tomcat simplifies server setup. Servlets and JSPs enable dynamic, scalable apps.

5. Best Practices

Keep controllers lightweight; delegate logic to services or DAOs (Data Access Objects).
Avoid heavy Java code in JSPs; use EL (${}) or JSTL for clean views.
Validate inputs in servlets to prevent errors or security issues (e.g., SQL injection).
Use a build tool (Maven/Gradle) to manage dependencies and create .war files.
Test with embedded Tomcat during development for fast iterations.
