Transitioning from Spring Core to Spring Boot

A Comprehensive Guide to Using Annotations and Layered Architecture

1. Introduction to Spring Core vs. Spring Boot

Spring Core: The foundational framework for dependency injection (DI) and inversion of control (IoC). It requires explicit configuration to manage beans, dependencies, and the application context.
Configurations can be XML-based or Java-based (using @Configuration classes).
Example: Defining a bean in Spring Core:
@Configuration
public class AppConfig {
    @Bean
    public Laptop laptop() {
        return new Laptop();
    }
}
Requires manual setup of the application context, e.g., AnnotationConfigApplicationContext.
Spring Boot: A framework built on Spring Core that simplifies development with an opinionated approach, reducing boilerplate code.
Uses the @SpringBootApplication annotation, which is a combination of:
  • @Configuration: Defines bean configurations.
  • @EnableAutoConfiguration: Automatically configures components based on classpath dependencies.
  • @ComponentScan: Scans for Spring components (e.g., @Component, @Service) in the package and sub-packages.
Automatically starts an embedded server (e.g., Tomcat for web applications) and initializes the application context.
Example:
@SpringBootApplication
public class SpringBootDemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(SpringBootDemoApplication.class, args);
    }
}

Key Differences

  • Configuration: Spring Core requires explicit configuration (XML or Java-based), while Spring Boot uses auto-configuration.
  • Setup: Spring Core needs manual context initialization, whereas Spring Boot handles it via SpringApplication.run().
  • Development Speed: Spring Boot reduces setup time, making it ideal for rapid development.
Added Point: Auto-Configuration in Spring Boot:
Spring Boot examines the classpath (e.g., dependencies in pom.xml) and automatically configures beans, such as a DataSource for database access if a database driver is present.
Example: Including spring-boot-starter-data-jpa automatically configures a JPA EntityManager.

2. Using Annotations in Spring Boot

Spring Boot leverages annotations to manage beans, inject dependencies, and configure properties, minimizing manual configuration.

Key Annotations

1. @Component:
Marks a class as a Spring-managed bean, making it eligible for component scanning.
Used for general-purpose components not specific to a layer (e.g., model classes).
How It Works: When Spring scans the package, it detects @Component classes, creates their instances, and stores them in the application context as beans.
Example:
@Component
public class Laptop implements Computer {
    public void compile() {
        System.out.println("Compiling in Laptop");
    }
}
Explanation: The Laptop class is scanned and instantiated as a bean. Spring manages its lifecycle and makes it available for dependency injection.
2. @Autowired:
Automatically injects dependencies into fields, setters, or constructors.
Resolves dependencies by type, but may throw an error if multiple beans of the same type exist.
How It Works: Spring searches the application context for a bean matching the type of the @Autowired field or parameter. If found, it injects the bean.
Example (Field Injection):
@Component
public class Alien {
    @Autowired
    private Computer com;
    public void code() {
        com.compile();
    }
}
Explanation: Spring injects a Computer bean (e.g., Laptop or Desktop) into the com field. The code() method calls compile() on the injected bean.
Example (Setter Injection):
@Component
public class Alien {
    private Computer com;
    @Autowired
    public void setCom(Computer com) {
        this.com = com;
    }
}
Explanation: Spring calls the setCom method, passing the Computer bean, allowing for more control (e.g., adding validation logic in the setter).
3. @Value:
Injects values from properties files, environment variables, or hard-coded defaults.
How It Works: Spring resolves the value (e.g., from application.properties or the annotation’s default) and assigns it to the field.
Example:
@Component
public class Alien {
    @Value("25")
    private int age;
    public int getAge() {
        return age;
    }
}
Explanation: The age field is set to 25. If a property like alien.age=30 exists in application.properties, Spring would use that value instead.
4. @Primary:
Specifies the default bean when multiple beans of the same type are available.
How It Works: When Spring encounters multiple beans of the same type, it prioritizes the @Primary bean unless overridden (e.g., by @Qualifier).
Example:
@Component
@Primary
public class Desktop implements Computer {
    public void compile() {
        System.out.println("Compiling in Desktop");
    }
}
Explanation: If both Laptop and Desktop implement Computer, Spring chooses Desktop for injection unless specified otherwise.
5. @Qualifier:
Resolves ambiguity by specifying the exact bean to inject, overriding @Primary.
How It Works: Spring uses the bean name provided in @Qualifier to select the correct bean from multiple candidates.
Example:
@Component
public class Alien {
    @Autowired
    @Qualifier("laptop")
    private Computer com;
    public void code() {
        com.compile();
    }
}
Explanation: Despite Desktop being @Primary, @Qualifier("laptop") ensures the Laptop bean is injected, so code() calls Laptop.compile().
6. @Service:
A stereotype annotation for service-layer classes that handle business logic.
Functionally equivalent to @Component but semantically indicates the class’s role.
How It Works: Like @Component, it registers the class as a bean, but it improves code readability by signaling that the class contains business logic.
Example:
@Service
public class LaptopService {
    public boolean addLaptop(Laptop laptop) {
        System.out.println("Method called");
        return true;
    }
}
Explanation: The LaptopService bean is created and managed by Spring. The addLaptop method simulates processing logic, such as validating or transforming the Laptop object.
7. @Repository:
A stereotype annotation for data access layer classes that interact with the database.
Provides additional features, such as exception translation for database operations.
How It Works: Registers the class as a bean and wraps database operations to convert low-level exceptions (e.g., SQL exceptions) into Spring’s DataAccessException.
Example:
@Repository
public class LaptopRepository {
    public void save(Laptop laptop) {
        System.out.println("Saved in database");
    }
}
Explanation: The LaptopRepository bean simulates database operations. The save method would typically execute SQL queries to persist the Laptop object.

Added Point: Bean Naming

By default, Spring uses the class name (lowercased) as the bean name (e.g., Laptop → laptop).
You can specify a custom name using the annotation’s value, e.g., @Component("customLaptop").
Example:
@Component("customLaptop")
public class Laptop implements Computer {
    public void compile() {
        System.out.println("Compiling in Laptop");
    }
}
Explanation: The Laptop bean is registered as customLaptop, so @Qualifier("customLaptop") would be used to inject it.

Example from GitHub Repository

Below is the complete code from the GitHub repository with detailed explanations of how each component works.
Alien.java: 
package com.telusko.model;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

@Component
public class Alien {
    @Value("25")
    private int age;
    private Computer com;

    public int getAge() {
        return age;
    }

    public void setAge(int age) {
        this.age = age;
    }

    public Computer getCom() {
        return com;
    }

    @Autowired
    @Qualifier("laptop")
    public void setCom(Computer com) {
        this.com = com;
    }

    public void code() {
        com.compile();
    }
}
How It Works:
  • @Component: Registers Alien as a Spring bean named alien.
  • @Value("25"): Sets age to 25.
  • @Autowired with @Qualifier("laptop") on setCom: Injects the Laptop bean (implementing Computer) into the com field via the setter.
  • code(): Calls compile() on the injected Computer bean, which outputs "Compiling in Laptop" due to @Qualifier("laptop").

  • @Component: Registers Desktop as a bean named desktop.
  • @Primary: Makes Desktop the default Computer bean if no @Qualifier is specified.
  • Without @Qualifier("laptop") in Alien, Desktop.compile() would be called, outputting "Compiling in Desktop".

How It Works:

  • Enables polymorphism, allowing Alien to work with any Computer implementation.

SpringBootDemoApplication.java: 
package com.telusko;

import com.telusko.model.Alien;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ApplicationContext;

@SpringBootApplication
public class SpringBootDemoApplication {
    public static void main(String[] args) {
        ApplicationContext context = SpringApplication.run(SpringBootDemoApplication.class, args);
        Alien obj = context.getBean(Alien.class);
        obj.code();
        System.out.println(obj.getAge());
    }
}
How It Works:
  • @SpringBootApplication: Initializes the Spring Boot application, scanning for components in com.telusko and sub-packages.
  • SpringApplication.run(): Starts the application, creating the application context.
  • context.getBean(Alien.class): Retrieves the Alien bean.
  • obj.code(): Calls code(), which invokes Laptop.compile() (due to @Qualifier("laptop")), outputting "Compiling in Laptop".
  • obj.getAge(): Outputs 25 (set by @Value("25")).
Output:
Compiling in Laptop
25
Explanation: The @Qualifier("laptop") ensures Laptop is injected into Alien, so code() calls Laptop.compile(). The age is printed as 25.

3. Layered Architecture in Spring Boot

Spring Boot applications follow a layered architecture to separate concerns, enhancing modularity and maintainability.

Layers

1. Client Layer:
Represents the end-user or external system (e.g., web browser, API client).
Sends requests to the server (e.g., HTTP requests in a web application).
2. Controller Layer:
Handles incoming requests and returns responses.
Delegates business logic to the Service layer.
Uses @Controller or @RestController (not used in this console-based example but relevant for web applications).
Added Point: In Spring MVC, controllers map URLs to methods using @RequestMapping or @GetMapping.
3. Service Layer:
Contains business logic, such as calculations, validations, or transformations.
Interacts with the Repository layer to fetch or persist data.
Marked with @Service.
Example: LaptopService processes laptop-related logic.
4. Repository Layer:
Handles database operations (e.g., CRUD operations).
Interacts with the database using technologies like JDBC, JPA, or Spring Data.
Marked with @Repository.
Example: LaptopRepository simulates database interactions.
5. Model Layer:
Represents domain entities (e.g., Laptop, Alien, Desktop).
Stored in a model package.
Marked with @Component or other annotations (e.g., @Entity for JPA).

Data Flow

Flow:
  1. Client sends a request to the Controller.
  2. Controller calls the Service layer for processing.
  3. Service layer interacts with the Repository layer to access data.
  4. Repository communicates with the Database.
  5. Data flows back: Database → Repository → Service → Controller → Client.
Example Flow (Based on the Code):
The SpringBootDemoApplication acts as the client, requesting an Alien bean.
The Alien bean depends on a Computer bean (Laptop in this case).
LaptopService (if used) would process the Laptop object and call LaptopRepository.
LaptopRepository simulates saving the Laptop to the database.
The result (e.g., "Compiling in Laptop" and age) is printed to the console.

Package Structure

Organizing classes in separate packages improves code organization:
  • com.telusko.model: Alien, Laptop, Desktop, Computer.
  • com.telusko.service: LaptopService.
  • com.telusko.repo: LaptopRepository.
Added Point: In real projects, additional packages like com.telusko.controller or com.telusko.config may be used for controllers and configurations.

4. Handling Bean Ambiguity

Problem: When multiple beans implement the same interface (e.g., Laptop and Desktop for Computer), Spring throws a NoUniqueBeanDefinitionException.
Solutions:
  1. @Primary: Marks one bean as the default.
    2. Example: @Primary on Desktop makes it the default Computer bean.
  2. @Qualifier: Specifies the exact bean to inject, overriding @Primary.
    3. Example: @Qualifier("laptop") selects the Laptop bean.
How It Works:
Without @Qualifier, Spring would inject Desktop (due to @Primary), printing "Compiling in Desktop".
With @Qualifier("laptop"), Spring injects Laptop, printing "Compiling in Laptop".
Added Point: Alternative to @Qualifier:
You can use @Resource (from JSR-250) to inject beans by name: 
@Resource(name = "laptop")
private Computer com;
Explanation: @Resource is similar to @Autowired with @Qualifier but is part of the Java EE standard.

5. Benefits of Spring Boot

  • Minimal Configuration: @SpringBootApplication handles component scanning and auto-configuration.
  • Embedded Server: Includes servers like Tomcat for web applications.
  • Simplified Dependency Management: Uses Spring Boot Starter dependencies (e.g., spring-boot-starter-web).
  • Property Files: Supports application.properties or application.yml for external configuration.
  • Added Point: Spring Boot Actuator:
    Provides production-ready features like health checks, metrics, and monitoring.
    Example: Add spring-boot-starter-actuator to enable endpoints like /actuator/health.

6. Additional Important Points

1. Bean Scopes:
Spring beans have scopes that define their lifecycle:
  • Singleton (default): One instance per application context.
  • Prototype: A new instance each time the bean is requested.
Example (Prototype Scope):
@Component
@Scope("prototype")
public class Laptop implements Computer {
    public void compile() {
        System.out.println("Compiling in Laptop");
    }
}
Explanation: Each call to context.getBean(Laptop.class) creates a new Laptop instance.
2. Application Properties:
Spring Boot allows external configuration via application.properties or application.yml.
Example (application.properties): 
alien.age=30
Inject using @Value("${alien.age}") or @ConfigurationProperties for complex configurations.
3. Error Handling:
If a bean is missing (e.g., LaptopRepository without @Repository), Spring throws a NoSuchBeanDefinitionException.
Use @Component, @Service, or @Repository to ensure beans are registered.
4. Spring Boot Starter Dependencies:
Simplify dependency management by bundling related libraries.
Example: spring-boot-starter-web includes Spring MVC, Tomcat, and JSON support.

8. Summary

  • Spring Core to Spring Boot: Spring Boot simplifies configuration with @SpringBootApplication, automating bean management and context initialization.
  • Annotations: @Component, @Autowired, @Value, @Primary, @Qualifier, @Service, and @Repository enable dependency injection and bean management with minimal configuration.
  • Layered Architecture: Separates concerns into Client, Controller, Service, Repository, and Model layers, improving modularity.
  • Example: The GitHub project demonstrates a console-based application with Alien, Laptop, Desktop, LaptopService, and LaptopRepository, showcasing annotations and layered architecture.
  • Diagrams: UML class diagram illustrates class relationships, while the layered architecture diagram shows data flow.
  • Added Value: Detailed code explanations and diagram instructions enhance understanding for beginners and intermediate developers.
