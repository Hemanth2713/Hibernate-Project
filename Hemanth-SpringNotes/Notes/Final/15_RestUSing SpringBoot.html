A Comprehensive Guide to Building REST APIs with Spring Boot

Introduction to REST with Spring Boot

Overview
REST (Representational State Transfer) is an architectural style for designing networked applications. In the context of Spring Boot, REST APIs allow a server to communicate with clients (browsers, mobile apps, or other servers) by sending and receiving data, typically in JSON or XML format, instead of rendering full HTML pages.
The goal is to create a backend that serves data to various frontends, such as React applications or mobile apps.
Why REST?
Separation of Concerns: Unlike traditional full-stack applications with tightly coupled frontend (e.g., JSP) and backend, REST APIs decouple the backend (data and logic) from the frontend (UI). This allows:
  • Independent development of frontend and backend.
  • Support for multiple clients (browsers, mobile apps, IoT devices).
Data-Centric: REST APIs return data (e.g., JSON) rather than rendered pages, making them lightweight and versatile.
Scalability: A single REST backend can serve multiple clients, reducing redundancy compared to maintaining separate servers for browsers and mobile apps.
Real-Time Example
Consider a job portal application:
  • Traditional Approach: A Spring Boot app with JSP renders HTML pages for job listings. Users access it via a browser, and the server returns fully formatted pages.
  • REST Approach: The same app exposes a REST API that returns job data in JSON. A React frontend or mobile app consumes this data to display jobs dynamically. For example:
    • Browser: A React app fetches job listings and renders them.
    • Mobile App: An iOS app fetches the same JSON data and displays it in a native UI.
    • Third-Party: A partner companyâ€™s server fetches job data for integration.
Project Context
The notes are based on a job portal application where:
  • Entities: Jobs (JobPost), Employees, Employers, Companies.
  • Operations: CRUD (Create, Read, Update, Delete) operations on job posts.
  • Goal: Transition from a JSP-based full-stack app to a REST API backend that serves JSON data to a React frontend or other clients.

What is REST?

Definition
REST (Representational State Transfer) is an architectural style for designing APIs where:
  • Resources: Data entities (e.g., JobPost) are treated as resources, identified by URLs (e.g., /jobposts).
  • State: The current state of a resource (e.g., a jobâ€™s details at a given time) is transferred between client and server.
  • Stateless: Each request from a client to the server must contain all necessary information. The server does not store client state between requests.
Key Principles
1. Resources as Nouns:
  • URLs represent resources, not actions (e.g., /jobposts instead of /getAllJobs).
  • Example: /jobposts/3 identifies a specific job with ID 3.
2. HTTP Methods:
  • REST uses standard HTTP methods to perform CRUD operations:
    • GET: Read data.
    • POST: Create data.
    • PUT: Update data.
    • DELETE: Delete data.
3. Stateless Communication:
  • Each request is independent, requiring authentication and context (e.g., user ID, token) every time.
4. Data Formats:
  • JSON (JavaScript Object Notation) is preferred for its simplicity and readability.
  • XML is supported but less common due to verbosity.
Real-Time Example
In the job portal:
  • Resource: A JobPost (e.g., a job listing for a Java Developer).
  • State: At a given time, the jobâ€™s state includes its ID, profile (e.g., "Java Developer"), description, experience, and tech stack.
  • Request: A client requests /jobposts/3 to get the current state of job ID 3.
  • Response: The server returns JSON:
    { "postId": 3, "profile": "Java Developer", "desc": "Looking for a Java developer", "exp": 5, "techs": ["Java", "Spring"] }

HTTP Methods

Overview
HTTP methods define the action to perform on a resource. In REST, the most commonly used methods map to CRUD operations:
  • GET: Retrieve a resource (Read).
  • POST: Create a new resource (Create).
  • PUT: Update an existing resource (Update).
  • DELETE: Delete a resource (Delete).
Usage in Spring Boot
Spring Boot provides annotations to map HTTP methods to controller methods:
  • @GetMapping: Handles GET requests.
  • @PostMapping: Handles POST requests.
  • @PutMapping: Handles PUT requests.
  • @DeleteMapping: Handles DELETE requests.
Real-Time Example
For the job portal:
  • GET /jobposts: Retrieves all job posts.
  • GET /jobposts/3: Retrieves job post with ID 3.
  • POST /jobposts: Creates a new job post.
  • PUT /jobposts/3: Updates job post with ID 3.
  • DELETE /jobposts/3: Deletes job post with ID 3.

Working with Postman

Purpose

Setup

    2. Create a new request by clicking the "+" button.
    3. Select an HTTP method (e.g., GET).

Creating a RestController

Purpose
A @RestController in Spring Boot handles HTTP requests and returns data (JSON/XML) instead of views. It replaces the traditional @Controller used for JSP rendering.
Steps
  1. Create a New Project:
    • Use Spring Initializr:
      • Group: com.co
      • Artifact: springbootvacation
      • Java: 21
      • Dependencies: Spring Web, Lombok
      • Packaging: JAR
    • Download and open in IntelliJ IDEA.
  2. Reuse Components:
    • Copy JobPost (model), JobRepo (repository), and JobService (service) from the old JSP project.
    • Delete unnecessary JSP views and properties.
  3. Create RestController:
    • Class: JobRestController
    • Annotation: @RestController
    • Inject JobService using @Autowired.
Code Example
package com.co.restcontroller; import com.co.model.JobPost; import com.co.service.JobService; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.web.bind.annotation.GetMapping; import org.springframework.web.bind.annotation.RestController; import java.util.List; @RestController public class JobRestController { @Autowired private JobService jobService; @GetMapping("/jobposts") public List getAllJobs() { return jobService.getAllJobs(); } }
Explanation:
  • @RestController: Combines @Controller and @ResponseBody, ensuring the response is serialized to JSON.
  • @GetMapping("/jobposts"): Maps GET requests to /jobposts to retrieve all job posts.
  • jobService.getAllJobs(): Fetches data from the service layer.
Testing
  • Browser: Visit http://localhost:8080/jobposts to see JSON output.
  • Postman: Send a GET request to http://localhost:8080/jobposts.
  • Result: JSON array of job posts.
Real-Time Example
  • Problem: The old JSP project returned a view (jobs.jsp). The new REST API should return JSON.
  • Solution: Use @RestController to return a List<JobPost> as JSON, tested via Postman or browser.

Code Example

Explanation:
  • @CrossOrigin(origins = "http://localhost:3000"): Allows requests from the React app running on port 3000.
Testing

  • Run the React app (npm start).
  • Verify that job posts are fetched from http://localhost:8080/jobposts and displayed.
Real-Time Example

  • Solution: Adding @CrossOrigin resolves the issue, allowing the React app to display jobs from the Spring Boot API.

Path Variable

Purpose
A path variable allows dynamic URL segments to identify specific resources (e.g., /jobposts/3 to get job ID 3).
Implementation
  • Use @PathVariable to capture URL segments.
  • Define the URL pattern with curly braces (e.g., /jobposts/{postId}).
Code Example
@GetMapping("/jobposts/{postId}") public JobPost getJob(@PathVariable("postId") int postId) { return jobService.getJob(postId); }
Explanation:
  • @GetMapping("/jobposts/{postId}"): Maps GET requests to /jobposts/{postId}.
  • @PathVariable("postId"): Captures the postId from the URL.
  • jobService.getJob(postId): Fetches the job from the service.
Repository Logic
public JobPost getJob(int postId) { for (JobPost job : jobs) { if (job.getPostId() == postId) { return job; } } return null; }
Explanation:
  • Iterates through the job list to find a job matching the postId.
  • Returns null if no match is found.
Testing
  • Postman: Send GET to http://localhost:8080/jobposts/3.
  • Result: JSON for job ID 3:
    { "postId": 3, "profile": "Java Developer", "desc": "Looking for a Java developer", "exp": 5, "techs": ["Java", "Spring"] }
Real-Time Example
  • Scenario: A user clicks a job listing (ID 3) in the React app.
  • Action: The app sends a GET request to /jobposts/3.
  • Result: The API returns the specific jobâ€™s details.

Sending Data and RequestBody

Purpose
Send data from the client to the server to create a new resource using a POST request and @RequestBody.
Implementation
  • Use @PostMapping to handle POST requests.
  • Use @RequestBody to deserialize JSON data into a Java object.
Code Example
@PostMapping("/jobposts") public JobPost addJob(@RequestBody JobPost jobPost) { jobService.addJob(jobPost); return jobService.getJob(jobPost.getPostId()); }
Explanation:
  • @PostMapping("/jobposts"): Maps POST requests to /jobposts.
  • @RequestBody JobPost jobPost: Deserializes the JSON payload into a JobPost object.
  • jobService.addJob(jobPost): Adds the job to the repository.
  • jobService.getJob(jobPost.getPostId()): Returns the newly added job.
Service Logic
public void addJob(JobPost jobPost) { jobRepo.addJob(jobPost); }
Repository Logic
public void addJob(JobPost jobPost) { jobs.add(jobPost); }
Testing
  • Postman:
    • URL: http://localhost:8080/jobposts
    • Method: POST
    • Body: Raw JSON:
      { "postId": 6, "profile": "iOS Developer", "desc": "Mobile dev for iOS", "exp": 2, "techs": ["Swift"] }
    • Response: JSON of the created job.
  • Verification: Send a GET to /jobposts to confirm the new job is added.
Real-Time Example
  • Scenario: An employer submits a new job listing via a form in the React app.
  • Action: The app sends a POST request with JSON data to /jobposts.
  • Result: The job is added, and the API returns the created job.

Put and Delete Mapping

Purpose
Handle updates (PUT) and deletions (DELETE) of resources.
PUT Mapping
Updates an existing resource using @PutMapping.
Code Example
@PutMapping("/jobposts") public JobPost updateJob(@RequestBody JobPost jobPost) { jobService.updateJob(jobPost); return jobService.getJob(jobPost.getPostId()); }
Explanation:
  • @PutMapping("/jobposts"): Maps PUT requests to /jobposts.
  • @RequestBody JobPost jobPost: Deserializes the updated JSON data.
  • jobService.updateJob(jobPost): Updates the job in the repository.
  • jobService.getJob(jobPost.getPostId()): Returns the updated job.
Repository Logic
public void updateJob(JobPost jobPost) { for (JobPost jobPost1 : jobs) { if (jobPost1.getPostId() == jobPost.getPostId()) { jobPost1.setPostProfile(jobPost.getPostProfile()); jobPost1.setDesc(jobPost.getDesc()); jobPost1.setExp(jobPost.getExp()); jobPost1.setTechs(jobPost.getTechs()); break; } } }
Explanation:
  • Finds the job by postId and updates its fields.
Testing
  • Postman:
    • URL: http://localhost:8080/jobposts
    • Method: PUT
    • Body: Raw JSON:
      { "postId": 2, "profile": "React Developer", "desc": "Looking for a React developer", "exp": 2, "techs": ["React", "JavaScript"] }
    • Response: Updated job JSON.
  • Verification: Send GET to /jobposts to confirm the update.
DELETE Mapping
Deletes a resource using @DeleteMapping.
Code Example
@DeleteMapping("/jobposts/{postId}") public String deleteJob(@PathVariable("postId") int postId) { jobService.deleteJob(postId); return "Deleted"; }
Explanation:
  • @DeleteMapping("/jobposts/{postId}"): Maps DELETE requests to /jobposts/{postId}.
  • @PathVariable("postId"): Captures the postId from the URL.
  • jobService.deleteJob(postId): Deletes the job.
  • Returns a confirmation message.
Repository Logic
public void deleteJob(int postId) { for (JobPost jobPost : jobs) { if (jobPost.getPostId() == postId) { jobs.remove(jobPost); break; } } }
Explanation:
  • Finds and removes the job by postId.
  • Uses break to avoid concurrent modification issues.
Testing
  • Postman:
    • URL: http://localhost:8080/jobposts/3
    • Method: DELETE
    • Response: "Deleted"
  • Verification: Send GET to /jobposts to confirm the job is removed.
Real-Time Example
  • PUT Scenario: An employer updates a job listing to change the profile from "Frontend Developer" to "React Developer".
  • DELETE Scenario: An employer removes a job listing (ID 3) that is no longer active.

Content Negotiation

Purpose
Content negotiation allows clients to specify the desired response format (e.g., JSON or XML) and the server to enforce specific formats.
Implementation
  • Default: Spring Boot uses the Jackson library to serialize Java objects to JSON.
  • XML Support: Add the jackson-dataformat-xml dependency for XML serialization.
  • Produces/Consumes: Use produces and consumes attributes in mappings to restrict formats.
Adding XML Support
com.fasterxml.jackson.dataformat jackson-dataformat-xml 2.15.3
Explanation:
  • Matches the Jackson version used for JSON (e.g., 2.15.3).
  • Enables XML serialization/deserialization.
Restricting Formats
Example with produces:
@GetMapping(value = "/jobposts", produces = "application/json") public List getAllJobs() { return jobService.getAllJobs(); }
Explanation:
  • produces = "application/json": Ensures only JSON is returned.
  • A request for XML (Accept: application/xml) will result in a 406 Not Acceptable error.
Example with consumes:
@PostMapping(value = "/jobposts", consumes = "application/xml") public JobPost addJob(@RequestBody JobPost jobPost) { jobService.addJob(jobPost); return jobService.getJob(jobPost.getPostId()); }
Explanation:
  • consumes = "application/xml": Accepts only XML input.
  • A JSON request will result in a 415 Unsupported Media Type error.
Testing
  • JSON Request:
    • Postman: Set Accept: application/json in headers.
    • URL: http://localhost:8080/jobposts
    • Result: JSON response.
  • XML Request:
    • Postman: Set Accept: application/xml in headers.
    • URL: http://localhost:8080/jobposts
    • Result: XML response (after adding jackson-dataformat-xml).
  • Restricted JSON:
    • Update @GetMapping with produces = "application/json".
    • Send XML request: Results in 406 error.
Real-Time Example
  • Scenario: A legacy system requires XML data, while modern clients use JSON.
  • Solution: Add jackson-dataformat-xml to support both formats, or use produces to enforce JSON for modern clients.

    • @PathVariable for dynamic URLs.
    • @RequestBody for deserializing input.
    • @CrossOrigin for frontend integration.
  • Tools:
    • Postman: Test APIs without a frontend.
    • React: Build a dynamic frontend.
  • Content Negotiation: Support JSON/XML with Jackson libraries and restrict formats using produces/consumes.
