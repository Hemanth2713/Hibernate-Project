Spring Java-Based Configuration Notes

Comprehensive guide to configuring Spring applications using Java-based configuration, based on Telusko's video.

1. Introduction to Java-Based Configuration

Purpose: Java-based configuration allows defining Spring beans using Java code instead of XML configuration files.
Why Use It:
Preferred by developers who dislike XML verbosity.
Provides type safety and better IDE support.
Offers programmatic control over bean creation.
Comparison with XML:
XML uses <bean> tags in a file (e.g., spring.xml).
Java-based configuration uses Java classes with annotations like @Configuration and @Bean.
When to Use:
Use XML if the project mandates it.
Use Java-based configuration for flexibility or preference.
Annotation-based configuration is another option (covered later).
Flexibility in Hybrid Configurations:
Java-based and XML configurations can coexist in the same project.
Example: Import XML config into a Java config class using @ImportResource("classpath:spring.xml").
Example:
XML Approach: ClassPathXmlApplicationContext loads spring.xml.
Java Approach: AnnotationConfigApplicationContext loads a Java config class.

2. Setting Up Java-Based Configuration

Steps:
  1. Create a package (e.g., com.telusko.config).
  2. Create a Java configuration class (e.g., AppConfig).
  3. Annotate the class with @Configuration.
  4. Define beans using @Bean annotations.
  5. Use AnnotationConfigApplicationContext to load the configuration.
Example:
AppConfig.java
package com.telusko.config;

import com.telusko.Desktop;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Bean;

@Configuration
public AppConfig {
    @Bean
    public Desktop desktop() {
        return new Desktop();
    }
}
App.java
package com.telusko;

import com.telusko.config.AppConfig;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public App {
    public static void main(String[] args) {
        AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
        Desktop dt = context.getBean(Desktop.class);
        dt.compile();
        context.close();
    }
}
Output:
Compiling using Desktop
Key Points:
@Configuration: Marks the class as a source of bean definitions.
@Bean: Defines a Spring-managed bean.
Spring manages the lifecycle (creation, injection, destruction) of beans, even though new is used in the code.
Error if no bean is defined: No qualifying bean of type 'Desktop' available.
Proxying with @Configuration:
@Configuration classes are enhanced by Spring’s CGLIB proxying to ensure bean methods are intercepted for singleton scope enforcement.
Avoid calling @Bean methods directly within the same class; use dependency injection instead.

3. Bean Names

Default Bean Name:
The method name in the @Bean annotation becomes the default bean name.
Example: public Desktop desktop() → Bean name is desktop.
Custom Bean Name:
Use the name attribute in @Bean to specify one or more names.
Example: @Bean(name = {"com2", "desktop1", "Beast"}).
Accessing Beans by Name:
Use context.getBean("beanName", Class.class) to retrieve a bean by its name.
If the name is incorrect, an error occurs: No bean named 'com2' available.
Bean Aliases:
Multiple names for a single bean act as aliases, allowing flexible bean retrieval.
Example: A bean named com2 can also be accessed as desktop1 or Beast.
Example:
@Configuration
public AppConfig {
    @Bean(name = {"com2", "desktop1", "Beast"})
    public Desktop desktop() {
        return new Desktop();
    }
}
Main Application
Desktop dt = context.getBean("com2", Desktop.class);
dt.compile();
Output:
Compiling using Desktop
Using Default Name:
Desktop dt = context.getBean("desktop", Desktop.class);

4. Bean Scope

Default Scope: Singleton (one instance per container).
Prototype Scope: Creates a new instance for each request.
Annotation: Use @Scope to define the scope of a bean.
Other Scopes:
Request Scope: One instance per HTTP request (@Scope("request")).
Session Scope: One instance per HTTP session (@Scope("session")).
Requires @Scope(proxyMode = ScopedProxyMode.TARGET_CLASS) for proper proxying in singleton beans.
Example:
@Configuration
public AppConfig {
    @Bean
    @Scope("prototype")
    public Desktop desktop() {
        return new Desktop();
    }
}
Main Application
Desktop dt1 = context.getBean(Desktop.class);
Desktop dt2 = context.getBean(Desktop.class);
dt1.compile();
dt2.compile();
Output:
Desktop object created
Desktop object created
Compiling using Desktop
Compiling using Desktop
Key Points:
Singleton: One instance shared across requests.
Prototype: New instance for each getBean() call.
Equivalent to XML: <bean scope="prototype"/>.

5. Dependency Injection with @Autowired

Purpose: Automatically injects dependencies into beans.
Usage in Java Config:
Pass dependencies as method parameters in @Bean methods.
Spring resolves dependencies from the container.
Example:
Classes
public class Alien {
    private int age;
    private Computer com;

    public void setAge(int age) { this.age = age; }
    public int getAge() { return age; }
    public void setCom(Computer com) { this.com = com; }
    public void code() { com.compile(); }
}

public interface Computer {
    void compile();
}

public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}
Configuration
@Configuration
public AppConfig {
    @Bean
    public Desktop desktop() {
        return new Desktop();
    }

    @Bean
    public Alien alien(Computer com) {
        Alien obj = new Alien();
        obj.setAge(25);
        obj.setCom(com);
        return obj;
    }
}
Main Application
AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
Alien obj = context.getBean(Alien.class);
System.out.println(obj.getAge());
obj.code();
context.close();
Output:
25
Compiling using Desktop
Key Points:
Spring automatically injects a Computer bean (e.g., Desktop) into the alien method.
@Autowired is optional in newer Spring versions for constructor or method injection.
Error if no matching bean is found: No qualifying bean of type 'Computer' available.
Lazy Initialization:
Use @Lazy on @Bean or @Component to delay bean creation until first use.
Example: @Bean @Lazy public Desktop desktop() { ... }.
Reduces startup time for beans that are not immediately needed.

6. Handling Multiple Beans with @Primary and @Qualifier

Problem: When multiple beans implement the same interface (e.g., Computer), Spring cannot decide which to inject.
Solutions:
  1. Qualifier: Specify the exact bean by name.
  2. Primary: Mark one bean as the default choice.
Example:
Additional Class
public class Laptop implements Computer {
    public void compile() { System.out.println("Compiling using Laptop"); }
}
Configuration
@Configuration
public AppConfig {
    @Bean
    public Desktop desktop() {
        return new Desktop();
    }

    @Bean
    public Laptop laptop() {
        return new Laptop();
    }

    @Bean
    public Alien alien(@Qualifier("desktop") Computer com) {
        Alien obj = new Alien();
        obj.setAge(25);
        obj.setCom(com);
        return obj;
    }
}
Using @Primary
@Bean
@Primary
public Laptop laptop() {
    return new Laptop();
}
Main Application
Alien obj = context.getBean(Alien.class);
obj.code();
Output (with @Qualifier("desktop")): 
Compiling using Desktop
Output (with @Primary on Laptop and no @Qualifier): 
Compiling using Laptop
Key Points:
@Qualifier takes precedence over @Primary.
@Qualifier specifies the bean name (e.g., desktop or laptop).
@Primary makes a bean the default when no qualifier is specified.
Error if multiple beans exist without resolution: Expected single matching bean but found two.
Qualifier with Constructor Injection:
@Qualifier can be used with constructor parameters for explicit dependency resolution.
Example:
@Bean
public Alien alien(@Qualifier("desktop") Computer com) {
    Alien obj = new Alien();
    obj.setCom(com);
    return obj;
}

7. Stereotype Annotations (@Component, @ComponentScan)

Purpose: Simplify configuration by marking classes as Spring-managed beans without explicit @Bean definitions.
Annotations:
@Component: Marks a class as a Spring-managed bean.
@ComponentScan: Tells Spring to scan packages for @Component-annotated classes.
Specialized Stereotype Annotations:
@Service: Indicates a service-layer component (semantic specialization of @Component).
@Repository: Indicates a data access component, with additional exception translation.
@Controller: Indicates a web controller (used in Spring MVC).
Example: @Service public class AlienService { ... }.

@Component makes a class a Spring bean without needing @Bean.
@ComponentScan scans for @Component-annotated classes in specified packages.
Default bean name: Class name with the first letter lowercase (e.g., Alien → alien).
Custom bean name: @Component("com2").
Error if @ComponentScan is missing: No qualifying bean of type 'Alien' available.

8. Autowiring with @Autowired

Purpose: Automatically injects dependencies into @Component-annotated classes.
Injection Types:
  1. Field Injection:
    2. @Autowired
private Computer com;
  2. Constructor Injection:
    3. private Computer com;
@Autowired
public Alien(Computer com) {
    this.com = com;
}
  3. Setter Injection:
    4. private Computer com;
@Autowired
public void setCom(Computer com) {
    this.com = com;
}
Best Practices for Injection:
Prefer constructor injection for mandatory dependencies to ensure immutability and testability.
Use setter injection for optional dependencies or when dependencies may change.
Avoid field injection in production code due to difficulties in unit testing and hidden dependencies.
Example:

public class Alien {
    private int age;

    @Qualifier("desktop")

    public int getAge() { return age; }
    public void code() { com.compile(); }
}

@Component
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}

@Component
public class Laptop implements Computer {
    public void compile() { System.out.println("Compiling using Laptop"); }
}
Configuration
@Configuration
@ComponentScan(basePackages = "com.telusko")
public AppConfig {}
Output:
Compiling using Desktop
Key Points:
@Autowired searches for a matching bean by type.
Error if multiple beans match: Expected single matching bean but found two.
@Qualifier resolves ambiguity by specifying the bean name.
Setter or constructor injection is preferred over field injection for testability.

9. @Primary vs @Qualifier in @Component

Primary:
Marks a @Component or @Bean as the default choice.
Example:
@Component
@Primary
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}
Qualifier:
Specifies the exact bean name in @Autowired.
Takes precedence over @Primary.
Example:
@Component
@Primary
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}

@Component
public class Laptop implements Computer {
    public void compile() { System.out.println("Compiling using Laptop"); }
}

@Component
public class Alien {
    @Autowired
    @Qualifier("laptop")
    private Computer com;

    public void code() { com.compile(); }
}
Output:
Compiling using Laptop
Key Points:
@Qualifier overrides @Primary.
Use @Primary to avoid specifying @Qualifier everywhere.
Use @Qualifier for explicit control.
Multiple @Primary Error:
Defining multiple @Primary beans for the same type causes a configuration error.
Example: @Primary on both Desktop and Laptop will throw BeanDefinitionOverrideException.

10. @Scope and @Value Annotations

Scope:
Defines the lifecycle of a @Component or @Bean.
Example:
@Component
@Scope("prototype")
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}
Value:
Injects a value into a field, typically from a property file or hard-coded.
Example:
@Component
public class Alien {
    @Value("21")
    private int age;
    @Autowired
    @Qualifier("desktop")
    private Computer com;

    public int getAge() { return age; }
    public void code() { com.compile(); }
}
Main Application

System.out.println(obj.getAge());
obj.code();
Output:
21
Compiling using Desktop
Key Points:
@Scope("prototype"): Creates a new instance per request.
@Value: Allows injecting values (e.g., from application.properties).
Example with properties:
@Value("${alien.age}")
private int age;

Dynamic Value Injection:
Use @Value with SpEL (Spring Expression Language) for dynamic values.
Example: @Value("#{systemProperties['user.name']}") private String userName;.

11. Key Annotations Summary

Annotation Purpose
@Configuration Marks a class as a source of bean definitions.
@Bean Defines a Spring-managed bean in a @Configuration class.
@Component Marks a class as a Spring-managed bean (auto-detected by @ComponentScan).
@ComponentScan Scans packages for @Component-annotated classes.
@Autowired Automatically injects dependencies (field, constructor, or setter).
@Qualifier Specifies a bean by name to resolve ambiguity.
@Primary Marks a bean as the default choice when multiple beans match.
@Scope Defines the scope of a bean (e.g., singleton, prototype).
@Value Injects a value into a field (hard-coded or from properties).
@Lazy Delays bean initialization until first use.
@ImportResource Imports XML configuration into a Java config class.

12. Advantages of Java-Based Configuration

Type Safety: Compile-time checking of bean definitions.
Programmatic Control: Logic can be added to bean creation.
No XML Overhead: Eliminates verbose XML files.
Integration with Annotations: Works seamlessly with @Component, @Autowired, etc.
Refactoring Ease:
Java code is easier to refactor using IDE tools compared to XML, reducing configuration errors.

13. Common Errors and Fixes

  1. No Qualifying Bean:
    2. Cause: No bean defined or @ComponentScan missing.
    Fix: Add @Bean or @Component and ensure @ComponentScan includes the package.
  2. Multiple Beans Found:
    3. Cause: Multiple beans implement the same interface.
    Fix: Use @Qualifier or @Primary.
  3. NullPointerException:
    4. Cause: Dependency not injected.
    Fix: Use @Autowired or ensure dependency is defined.
  4. BeanCreationException:
    5. Cause: Circular dependencies or missing dependencies.
    Fix: Use @Lazy on one of the beans or refactor to break the cycle.

Use setter injection for optional dependencies or when dependencies may change.
Avoid field injection in production code due to difficulties in unit testing and hidden dependencies.
Example:
@Component
public class Alien {
    private int age;
    @Autowired
    @Qualifier("desktop")
    private Computer com;

    public void setAge(int age) { this.age = age; }
    public int getAge() { return age; }
    public void code() { com.compile(); }
}

@Component
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}

@Component
public class Laptop implements Computer {
    public void compile() { System.out.println("Compiling using Laptop"); }
}
Configuration
@Configuration
@ComponentScan(basePackages = "com.telusko")
public AppConfig {}
Output:
Compiling using Desktop
Key Points:
@Autowired searches for a matching bean by type.
Error if multiple beans match: Expected single matching bean but found two.
@Qualifier resolves ambiguity by specifying the bean name.
Setter or constructor injection is preferred over field injection for testability.

9. @Primary vs @Qualifier in @Component

Primary:
Marks a @Component or @Bean as the default choice.
Example:
@Component
@Primary
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}
Qualifier:
Specifies the exact bean name in @Autowired.
Takes precedence over @Primary.
Example:
@Component
@Primary
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}

@Component
public class Laptop implements Computer {
    public void compile() { System.out.println("Compiling using Laptop"); }
}

@Component
public class Alien {
    @Autowired
    @Qualifier("laptop")
    private Computer com;

    public void code() { com.compile(); }
}
Output:
Compiling using Laptop
Key Points:
@Qualifier overrides @Primary.
Use @Primary to avoid specifying @Qualifier everywhere.
Use @Qualifier for explicit control.
Multiple @Primary Error:
Defining multiple @Primary beans for the same type causes a configuration error.
Example: @Primary on both Desktop and Laptop will throw BeanDefinitionOverrideException.

10. @Scope and @Value Annotations

Scope:
Defines the lifecycle of a @Component or @Bean.
Example:
@Component
@Scope("prototype")
public class Desktop implements Computer {
    public void compile() { System.out.println("Compiling using Desktop"); }
}
Value:
Injects a value into a field, typically from a property file or hard-coded.
Example:
@Component
public class Alien {
    @Value("21")
    private int age;
    @Autowired
    @Qualifier("desktop")
    private Computer com;

    public int getAge() { return age; }
    public void code() { com.compile(); }
}
Main Application
Alien obj = context.getBean(Alien.class);
System.out.println(obj.getAge());
obj.code();
Output:
21
Compiling using Desktop
Key Points:
@Scope("prototype"): Creates a new instance per request.
@Value: Allows injecting values (e.g., from application.properties).
Example with properties:
@Value("${alien.age}")
private int age;
            
alien.age=21
Dynamic Value Injection:
Use @Value with SpEL (Spring Expression Language) for dynamic values.
Example: @Value("#{systemProperties['user.name']}") private String userName;.

11. Key Annotations Summary

Annotation Purpose
@Configuration Marks a class as a source of bean definitions.
@Bean Defines a Spring-managed bean in a @Configuration class.
@Component Marks a class as a Spring-managed bean (auto-detected by @ComponentScan).
@ComponentScan Scans packages for @Component-annotated classes.
@Autowired Automatically injects dependencies (field, constructor, or setter).
@Qualifier Specifies a bean by name to resolve ambiguity.
@Primary Marks a bean as the default choice when multiple beans match.
@Scope Defines the scope of a bean (e.g., singleton, prototype).
@Value Injects a value into a field (hard-coded or from properties).
@Lazy Delays bean initialization until first use.
@ImportResource Imports XML configuration into a Java config class.

12. Advantages of Java-Based Configuration

Type Safety: Compile-time checking of bean definitions.
Programmatic Control: Logic can be added to bean creation.
No XML Overhead: Eliminates verbose XML files.
Integration with Annotations: Works seamlessly with @Component, @Autowired, etc.
Refactoring Ease:
Java code is easier to refactor using IDE tools compared to XML, reducing configuration errors.

13. Common Errors and Fixes

  1. No Qualifying Bean:
    2. Cause: No bean defined or @ComponentScan missing.
    Fix: Add @Bean or @Component and ensure @ComponentScan includes the package.
  2. Multiple Beans Found:
    3. Cause: Multiple beans implement the same interface.
    Fix: Use @Qualifier or @Primary.
  3. NullPointerException:
    4. Cause: Dependency not injected.
    Fix: Use @Autowired or ensure dependency is defined.
  4. BeanCreationException:
    5. Cause: Circular dependencies or missing dependencies.
    Fix: Use @Lazy on one of the beans or refactor to break the cycle.

16. Additional Points

  1. Conditional Bean Creation:
    2. Use @Conditional or @Profile to create beans based on conditions or active profiles.
    Example:
    @Bean
@Profile("dev")
public Desktop desktop() {
    return new Desktop();
}
    Activate profile: -Dspring.profiles.active=dev.
  2. Bean Post-Processing:
    3. Implement BeanPostProcessor to customize bean initialization.
    Example: Log bean creation or modify properties before/after initialization.
  3. Testing Java-Based Configuration:
    4. Use @SpringBootTest or TestContext framework for unit/integration testing.
    Example:
    @RunWith(SpringRunner.class)
@ContextConfiguration(classes = AppConfig.class)
public class AppTest {
    @Autowired
    private Alien alien;
    @Test
    public void testAlien() {
        assertNotNull(alien);
    }
}
  4. Performance Considerations:
    5. Minimize @ComponentScan scope to specific packages to reduce startup time.
    Use @Lazy for non-critical beans to optimize resource usage.
  5. Migration from XML to Java Config:
    6. Gradually replace XML beans with @Bean definitions.
    Use @ImportResource to bridge XML and Java configurations during migration.
  6. Spring Boot Integration:
    7. Java-based configuration is the foundation for Spring Boot’s auto-configuration.
    Spring Boot reduces manual configuration by providing @EnableAutoConfiguration.
    Example: @SpringBootApplication combines @Configuration, @ComponentScan, and @EnableAutoConfiguration.
