Spring Security Notes

For Beginners and Intermediate Learners

Spring Security is like a security guard for your web application, ensuring only the right people access the right resources. This guide covers 22 key topics, including Bcrypt encoding and database-driven authentication, explained with real-world analogies, code examples, and key points in a Telusko-style, engaging format!

1. Importance of Security in Application Development

Concept Overview

Security in application development is like locking the doors of your house to keep intruders out. Developers prioritize functionality (making the app work), stability (handling errors), and performance (fast responses). Security ensures only authorized users access features and data, using authentication (verifying identity) and authorization (controlling permissions). For sensitive data like credit card or medical records, security is critical to prevent breaches and comply with regulations.

Key Points

  • Functionality: Ensures the app works (e.g., job search in a portal).
  • Stability: Handles exceptions for consistent performance.
  • Performance: Delivers fast responses across features.
  • Security: Protects data and restricts access.
  • Authentication: Verifies user identity.
  • Authorization: Defines user permissions.
  • Data Sensitivity: Critical for personal or financial data.
  • Compliance: Meets legal standards like GDPR or HIPAA.

Example

In a job portal, anyone can view listings, but only authenticated admins can create or delete jobs. Spring Security enforces these restrictions.

Live Example

Without security, localhost:8080/admin/createJob allows anyone to post jobs. Spring Security requires admin login for this URL.

2. OWASP Top 10 Security Risks

Concept Overview

The OWASP Top 10 is like a “Most Wanted” list of hacker tactics, updated every four years (last in 2021). It lists the top 10 web application vulnerabilities. Spring Security helps mitigate these through secure authentication and authorization.

Key Risks

Risk Description
Broken Access ControlUsers access restricted resources.
Cryptographic FailuresWeak encryption exposes data.
InjectionMalicious code in inputs.
Insecure DesignFlawed app design creates vulnerabilities.
Security MisconfigurationDefault settings expose weaknesses.
Vulnerable ComponentsOld libraries with issues.
Authentication FailuresWeak passwords or no MFA.
Data Integrity FailuresUnverified data compromises app.
Logging FailuresMissing logs to detect attacks.
SSRFServer makes unauthorized requests.

Example

A SQL injection attack (e.g., admin' OR '1'='1) could bypass login. Spring Security’s prepared statements prevent this.

Key Points

  • OWASP Top 10 prioritizes critical security fixes.
  • Spring Security addresses risks like injection and broken access control.
  • Validate inputs and use secure configurations.
  • Regularly update dependencies to avoid vulnerable components.

3. Creating a Spring Security Project

Concept Overview

Creating a Spring Security project is like installing a security system. Adding the Spring Security dependency provides a default login page to protect all URLs.

Steps

  1. Visit Spring Initializr.
  2. Select Maven, Java 21, and add Spring Web, Security, DevTools.
  3. Set group ID com.co, artifact ID spring-sec-demo.
  4. Generate, unzip, and open in an IDE.
  5. Create a REST controller.

Code Example: HelloController

// Controller to handle HTTP requests for demo endpoints
package com.co;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

// Marks this class as a REST controller to handle HTTP requests

    @GetMapping("/hello")
    public String greet() {
        return "Hello World";
    }

    // Maps GET requests to /about, returns a brand name
    @GetMapping("/about")
    public String about() {
        return "Telusko";
    }

Live Example

Without Spring Security, localhost:8080/hello shows “Hello World”. With the dependency, it prompts a login form.

  • Spring Security dependency auto-enables security.
  • All URLs are protected by default.
  • Use Spring Initializr for quick setup.
  • Add DevTools for live reloading during development.

5. Spring Security Filters

Concept Overview

Spring Security filters are like airport checkpoints, intercepting requests before they reach the application code.

Request Flow

  1. Client Sends Request: User visits localhost:8080/hello.
  2. Servlet Container Receives: Hits server (e.g., Tomcat).
  3. Filter Chain: Spring Security filters intercept.
  4. Filters Process: Check authentication, CSRF, etc.
  5. Servlet: If filters pass, reaches controller.
  6. Response: Goes through filters to client.

Key Filters

  • SecurityContextPersistenceFilter: Stores user details.
  • UsernamePasswordAuthenticationFilter: Verifies credentials.
  • CsrfFilter: Ensures CSRF token for POST requests.
  • LogoutFilter: Handles logout.

Live Example

A POST request without a CSRF token fails (403), showing CsrfFilter in action.

Key Points

  • Filters execute in a specific order.
  • Each filter has a unique role.
  • Failure in any filter blocks the request.
  • Custom filters can be added for specific needs.
  • Filter misconfiguration can lead to security gaps.

6. Session Management and Session ID

Concept Overview

A session is like a visitor pass, with a unique session ID stored in a cookie (JSESSIONID) for authentication.

Code Example: Printing Session ID

// Controller to demonstrate session ID handling
package com.co;

import jakarta.servlet.http.HttpServletRequest;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

// Marks this class as a REST controller
@RestController
public class HelloController {
    // Maps GET requests to /hello, returns greeting with session ID
    @GetMapping("/hello")
    public String greet(HttpServletRequest request) {
        // Retrieves session ID from the request
        return "Hello World " + request.getSession().getId();
    }

    // Maps GET requests to /about, returns brand with session ID
    @GetMapping("/about")
    public String about(HttpServletRequest request) {
        // Same session ID is reused within the same session
        return "Telusko " + request.getSession().getId();
    }
}

Live Example

localhost:8080/hello shows “Hello World 1234”. Logging out and back in generates a new ID.

Key Points

  • Session ID is unique per session.
  • Stored in JSESSIONID cookie.
  • Logout invalidates the session.
  • Use secure cookies in production (HTTPS).

9. Cross-Site Request Forgery (CSRF)

Concept Overview

CSRF is like a malicious website tricking your browser into using your session ID to access a secure site (e.g., a banking app). Spring Security mitigates this by requiring a CSRF token for non-GET requests (POST, PUT, DELETE).

Key Points

  • CSRF tokens prevent unauthorized requests.
  • Enabled by default for non-GET requests.
  • Malicious sites can’t access tokens, enhancing security.
  • Disable CSRF only for stateless APIs (e.g., JWT).
  • Include tokens in forms or AJAX headers.

Live Example

A POST request to add a student without a CSRF token fails (403). Including the token succeeds.

10. CSRF Token in Action

Concept Overview

CSRF tokens are like one-time tickets for secure actions. Spring Security requires them for POST, PUT, and DELETE requests to prevent CSRF attacks.

Code Example: StudentController

// Controller to manage student data with CSRF-protected endpoints
package com.co;

import lombok.Data;
import org.springframework.web.bind.annotation.*;

import java.util.ArrayList;
import java.util.List;

// Marks this class as a REST controller
@RestController
public class StudentController {
    // In-memory list to store students
    private List students = new ArrayList<>(List.of(
        new Student(1, "Naveen", "Java"),
        new Student(2, "Kiran", "Blockchain")
    ));

    // Maps GET requests to /students, returns all students
    @GetMapping("/students")
    public List getStudents() {
        return students;
    }

    // Maps POST requests to /students, adds a student (CSRF token required)
    @PostMapping("/students")
    public void addStudent(@RequestBody Student student) {
        students.add(student);
    }
}

// Data class for student entity
@Data
class Student {
    private int id;
    private String name;
    private String technology;

    // Constructor for initializing student objects
    public Student(int id, String name, String technology) {
        this.id = id;
        this.name = name;
        this.technology = technology;
    }
}

Live Example

A POST to /students without a CSRF token fails (403). Sending the token in the header (X-CSRF-TOKEN) succeeds.

Key Points

  • CSRF tokens are unique per session.
  • Required for stateful apps with sessions.
  • Retrieve tokens via a GET endpoint (e.g., /csrf-token).
  • Test CSRF protection using tools like Postman.

11. Retrieving CSRF Token

Concept Overview

Retrieving a CSRF token is like getting a passcode for a secure transaction. Spring Security provides tokens via request attributes, accessible in controllers.

Code Example: Get CSRF Token

// Maps GET requests to /csrf-token, returns the CSRF token
@GetMapping("/csrf-token")
public CsrfToken getCsrfToken(HttpServletRequest request) {
    // Retrieves CSRF token from request attributes
    return (CsrfToken) request.getAttribute("_csrf");
}

Live Example

A GET to /csrf-token returns a token. Including it in a POST request’s X-CSRF-TOKEN header allows the request to succeed.

Key Points

  • Tokens are stored in request attributes (_csrf).
  • Use in headers for non-GET requests.
  • Enhances security against CSRF attacks.
  • Expose tokens securely to front-end apps.

13. Custom Security Configuration

Concept Overview

Custom security configuration is like setting your own house rules. It allows disabling CSRF, enabling stateless sessions, or configuring authentication methods.

Code Example: SecurityConfig

// Configuration class for custom Spring Security settings
package com.co.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.web.SecurityFilterChain;

// Enables Spring Security and allows custom configuration
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    // Defines the security filter chain for HTTP requests
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // Disables CSRF for stateless APIs (use with caution)
            .csrf(csrf -> csrf.disable())
            // Requires authentication for all requests
            .authorizeHttpRequests(request -> request.anyRequest().authenticated())
            // Enables form-based login with custom success URL
            .formLogin(form -> form.defaultSuccessUrl("/hello"))
            // Enables Basic Authentication with a custom realm
            .httpBasic(basic -> basic.realmName("my-realm"))
            // Configures stateless session management (no session storage)
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
        // Builds and returns the security filter chain
        return http.build();
    }
}

Live Example

Disabling CSRF allows POST requests without tokens. Stateless sessions generate new session IDs per request.

Key Points

  • Customizes default Spring Security settings.
  • Lambda syntax simplifies configuration.
  • Stateless sessions eliminate CSRF token need.
  • Use CSRF protection for stateful apps.
  • Test configurations thoroughly to avoid gaps.

14. Imperative Security Configuration

Concept Overview

Imperative configuration is like manually wiring a security system. It uses explicit objects and methods, making the process clearer but more verbose than lambda syntax.

Code Example: Imperative SecurityConfig

// Configuration class for imperative Spring Security setup
package com.co.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.Customizer;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.web.SecurityFilterChain;

// Enables Spring Security for custom configuration
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    // Defines the security filter chain explicitly
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // Customizer to disable CSRF
        Customizer> custCsrf = new Customizer>() {
            @Override
            public void customize(CsrfConfigurer configurer) {
                configurer.disable();
            }
        };

        // Customizer to require authentication for all requests
        Customizer.AuthorizationManagerRequestMatcherRegistry> custHttp = new Customizer.AuthorizationManagerRequestMatcherRegistry>() {
            @Override
            public void customize(AuthorizeHttpRequestsConfigurer.AuthorizationManagerRequestMatcherRegistry registry) {
                registry.anyRequest().authenticated();
            }
        };

        return http
            // Applies CSRF customizer
            .csrf(custCsrf)
            // Applies HTTP request authorization
            .authorizeHttpRequests(custHttp)
            // Configures stateless session management
            .sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            .and()
            // Enables Basic Authentication
            .httpBasic(Customizer.withDefaults())
            // Builds the security filter chain
            .build();
    }
}

Live Example

The imperative approach achieves the same result as lambda but is more explicit, showing how CSRF and authentication are configured.

Key Points

  • Verbose but clearer for understanding.
  • Uses builder pattern for configuration.
  • Lambda is preferred in production.
  • Ensure all customizers are correctly implemented.

15. Hardcoding Multiple Users

Concept Overview

Hardcoding multiple users is like issuing guest passes with predefined credentials. It uses an in-memory user store for testing multiple users and roles.

Code Example: InMemoryUserDetailsManager

// Configuration class to define in-memory users
package com.co.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;

// Defines Spring configuration for user details
@Configuration
public class SecurityConfig {

    // Creates an in-memory user store
    @Bean
    public UserDetailsService userDetailsService() {
        // Defines a user with username, password, and role
        var user = User
            .withDefaultPasswordEncoder() // Deprecated, for demo only
            .username("naveen")
            .password("n@123")
            .roles("USER")
            .build();

        // Defines an admin user with different credentials
        var admin = User
            .withDefaultPasswordEncoder() // Not for production
            .username("admin")
            .password("admin@789")
            .roles("ADMIN")
            .build();

        // Returns an in-memory user store with both users
        return new InMemoryUserDetailsManager(user, admin);
    }
}

Live Example

Login with naveen/n@123 or admin/admin@789 works; application properties credentials are ignored.

Key Points

  • InMemoryUserDetailsManager stores users in memory.
  • Deprecated withDefaultPasswordEncoder for learning only.
  • Supports multiple users and roles.
  • Use database for production user management.

16. Database-Driven Authentication

Concept Overview

Database-driven authentication is like checking a guest list in a database. It uses a custom UserDetailsService to fetch credentials from a database, secured by a custom authentication provider.

Code Example: User, UserPrincipal, MyUserDetailsService, UserRepo

// JPA entity for user data
package com.co.model;

import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Table;
import lombok.Data;

// Marks this class as a JPA entity
@Data
@Entity
@Table(name = "users")
public class User {
    @Id
    private int id; // Unique identifier
    private String username; // Login username
    private String password; // Stored password (hashed in production)
}

// UserDetails implementation for Spring Security
package com.co.model;

import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.userdetails.UserDetails;

import java.util.Collection;
import java.util.Collections;

// Adapts User entity to Spring Security’s UserDetails
public class UserPrincipal implements UserDetails {
    private final User user;

    public UserPrincipal(User user) {
        this.user = user;
    }

    // Defines user roles/authorities
    @Override
    public Collection getAuthorities() {
        return Collections.singleton(new SimpleGrantedAuthority("USER"));
    }

    @Override
    public String getPassword() {
        return user.getPassword();
    }

    @Override
    public String getUsername() {
        return user.getUsername();
    }

    // Account status methods (hardcoded for simplicity)
    @Override
    public boolean isAccountNonExpired() {
        return true;
    }

    @Override
    public boolean isAccountNonLocked() {
        return true;
    }

    @Override
    public boolean isCredentialsNonExpired() {
        return true;
    }

    @Override
    public boolean isEnabled() {
        return true;
    }
}

// Service to load user details from database
package com.co.service;

import com.co.dao.UserRepo;
import com.co.model.User;
import com.co.model.UserPrincipal;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.stereotype.Service;

// Implements UserDetailsService for database-driven authentication
@Service
public class MyUserDetailsService implements UserDetailsService {

    @Autowired
    private UserRepo userRepo; // JPA repository for user data

    // Loads user by username, throws exception if not found
    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userRepo.findByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("User not found: " + username);
        }
        return new UserPrincipal(user);
    }
}

// JPA repository for user data access
package com.co.dao;

import com.co.model.User;
import org.springframework.data.jpa.repository.Jparepository;

// Provides CRUD operations and custom query for username
public interface UserRepo extends JpaRepository {
    User findByUsername(String username);
}

Live Example

Login with database credentials (e.g., naveen/n@789) succeeds; invalid usernames throw a UsernameNotFoundException.

Key Points

  • Uses JPA for database access.
  • UserPrincipal bridges entity and UserDetails.
  • Requires database configuration in application.properties.
  • Add indexes to username for faster queries.

17. Configuring DaoAuthenticationProvider

Concept Overview

Configuring DaoAuthenticationProvider is like setting up a security checkpoint that verifies credentials against a database using a custom UserDetailsService.

Code Example: SecurityConfig with DaoAuthenticationProvider

// Configuration class for authentication provider
package com.co.config;

import com.co.service.MyUserDetailsService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationProvider;
import org.springframework.security.authentication.dao.DaoAuthenticationProvider;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.web.SecurityFilterChain;

// Enables Spring Security
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Autowired
    private UserDetailsService userDetailsService; // Custom user details service

    // Configures DaoAuthenticationProvider for database authentication
    @Bean
    public AuthenticationProvider authProvider() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService); // Sets user details service
        provider.setPasswordEncoder(NoOpPasswordEncoder.getInstance()); // No encoding (insecure)
        return provider;
    }

    // Defines security filter chain
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Disables CSRF for demo
            .authorizeHttpRequests(request -> request.anyRequest().authenticated()) // All requests need auth
            .httpBasic(basic -> basic.realmName("my-realm")) // Enables Basic Auth
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)); // Stateless
        return http.build();
    }
}

Live Example

The provider authenticates users via MyUserDetailsService, fetching credentials from the database.

Key Points

  • Requires UserDetailsService and password encoder.
  • NoOpPasswordEncoder is insecure for production.
  • Integrates with JPA repository for data access.
  • Use Bcrypt for secure password storage.

18. Introduction to Bcrypt for Password Security

Concept Overview

Bcrypt is like a super-strong safe for passwords. Unlike plain-text storage, which exposes passwords to attackers, Bcrypt uses hashing—a one-way process that generates a unique, irreversible string. It applies multiple rounds (e.g., 2^12) to make brute-force attacks computationally expensive, ensuring security even if the database is compromised.

Key Features

  • Hashing: One-way, no decryption possible.
  • Rounds: Configurable iterations (e.g., 10 or 12) for added security.
  • Format: Starts with $2a (Bcrypt version) and $12 (rounds).

Live Example

Encoding h@123 with Bcrypt (12 rounds) produces a string like $2a$12$..., stored securely in the database.

Key Points

  • Bcrypt is more secure than MD5 or SHA-256.
  • Prevents password exposure in breaches.
  • Requires matching rounds for encoding and verification.
  • Higher rounds increase security but slow performance.
  • Use at least 10 rounds for modern hardware.

20. Bcrypt Encoding for User Registration

Concept Overview

Bcrypt encoding during registration is like locking a password in a vault. Before saving a user, the plain-text password is hashed with Bcrypt, ensuring secure storage.

Code Example: UserService with Bcrypt

// Service to handle user registration with Bcrypt
package com.co.service;

import com.co.dao.UserRepo;
import com.co.model.User;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.stereotype.Service;

// Marks this class as a service component
@Service
public class UserService {

    @Autowired
    private UserRepo userRepo; // JPA repository for user data

    // Saves user with Bcrypt-encoded password
    public User saveUser(User user) {
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12); // 12 rounds for security
        String encodedPassword = encoder.encode(user.getPassword()); // Hashes password
        System.out.println(encodedPassword); // Debugging only, remove in production
        user.setPassword(encodedPassword); // Sets hashed password
        return userRepo.save(user); // Saves user to database
    }
}

Live Example

Registering avni with a@123 stores a Bcrypt hash (e.g., $2a$12$...) in the database.

Key Points

  • Uses BCryptPasswordEncoder with 12 rounds.
  • Passwords are hashed before storage.
  • Console logging is for debugging only.
  • Reuse encoder bean for consistency.

21. Configuring Bcrypt for Authentication

Concept Overview

Configuring Bcrypt for authentication is like upgrading a lock to match the vault. The DaoAuthenticationProvider is updated to use BCryptPasswordEncoder, ensuring secure password verification.

Code Example: SecurityConfig with Bcrypt

// Configuration class for Bcrypt-based authentication
package com.co.config;

import com.co.service.MyUserDetailsService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationProvider;
import org.springframework.security.authentication.dao.DaoAuthenticationProvider;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.web.SecurityFilterChain;

// Enables Spring Security
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Autowired
    private UserDetailsService userDetailsService; // Custom user details service

    // Configures DaoAuthenticationProvider with Bcrypt
    @Bean
    public AuthenticationProvider authProvider() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService); // Sets user details service
        provider.setPasswordEncoder(new BCryptPasswordEncoder(12)); // Bcrypt with 12 rounds
        return provider;
    }

    // Defines security filter chain
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Disables CSRF for demo
            .authorizeHttpRequests(request -> request.anyRequest().authenticated()) // All requests need auth
            .httpBasic(basic -> basic.realmName("my-realm")) // Enables Basic Auth
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)); // Stateless
        return http.build();
    }
}

Live Example

Login with avni/a@123 succeeds, as the database stores Bcrypt-encoded passwords. Plain-text passwords fail.

Key Points

  • Matches encoding strength (12 rounds) for registration and authentication.
  • Replaces NoOpPasswordEncoder for security.
  • Requires database update for existing users.
  • Use a single encoder bean for consistency.
  • Monitor performance with high round counts.

22. Project Organization and Best Practices

Concept Overview

Organizing a project is like arranging a filing cabinet. A clear package structure and best practices ensure maintainability and scalability.

Package Structure

  • controllers: HelloController, StudentController, UserController.
  • model: User, Student, UserPrincipal.
  • service: MyUserDetailsService, UserService.
  • dao: UserRepo.
  • config: SecurityConfig.

Code Example: application.properties

# Configures PostgreSQL database connection
spring.datasource.url=jdbc:postgresql://localhost:5432/springsecurity
# Database username
spring.datasource.username=postgres
# Database password
spring.datasource.password=admin
# Automatically updates database schema
spring.jpa.hibernate.ddl-auto=update
# Logs SQL queries for debugging
spring.jpa.show-sql=true
# Specifies PostgreSQL dialect for Hibernate
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.PostgreSQLDialect
